# AutoButton 初始项目研发技术总结与反思 (DEV_POSTMORTEM)

本文件详尽记录了 AutoButton 在**初始项目立项研发阶段**的技术栈选择与特点、对核心需求的盲区反思、AI Agent 开发同类系统工具的 Rule 宪法模版，以及对开发调试过程中多次重复返工的复盘。

---

## 📊 1. 初始立项技术栈与选择特点

在 AutoButton 初始立项阶段，开发团队为了快速跑通产品原型并打通核心自动化逻辑，采用了以下初始技术选型：

* **核心壳容器 (Shell)**：**Electron (Chromium + Node.js)**
* **物理按键模拟 (Input)**：**`robotjs`** (基于 C++ Node-API 二进制包)
* **数字识别 (OCR)**：**`tesseract.js`** (基于 WebAssembly 封装的离线图像引擎)
* **激活窗口追踪**：PowerShell 管道高频轮询
* **Mock 许可证服务**：本地常驻的 Node.js Express 服务端

### 💡 初始选型的合理考量（开发初期的权衡）
1. **开发敏捷度与跨平台便利**：Electron 允许使用成熟的现代前端技术栈（React + TS + TailwindCSS）快速堆叠出极其高精美、有科技感的拟态 UI，且排版布局开发效率极高。
2. **现成组件开箱即用**：不需要花费时间去手写操作系统级的 Win32 C++ / Rust 原生 APIs。`robotjs` 和 `tesseract.js` 在 Node 社区都是现成、成熟的自动化和识图包，开发成本极低。
3. **联调边界清晰**：通过本地另起 Express 服务器提供 License `/verify` 接口，使前后端在 Electron 环境下的网络联调结构十分清晰。

### ⚠️ 初始技术选型忽视的核心缺陷
虽然初始方案在短时间内实现了功能闭环，但却在关键的用户体验上引入了难以妥协的**重大硬伤**：
1. **包体体积失控与冷启动延迟**：Chromium 外壳加上 `tesseract.js` 内部打包的 80MB+ WebAssembly 图像语言训练集，导致绿色单文件体积堆积至 **93.6 MB**。这使得双击启动时解压零碎文件耗时超过 **1.5秒**，完全丧失了轻量、免安装、随开随用工具的便捷性。
2. **系统底层稳定性弱**：`robotjs` 的 64 位 C++ 驱动包与 Windows 1.5倍等非标准缩放屏幕结合时，频繁因为内联内存地址步进越界计算发生段错误（Segmentation fault）而引发程序频繁闪退；高频 PowerShell 进程轮询也触发了 Windows 杀毒软件的频繁拦截与误报。
3. **架构臃肿冗余**：仅仅为了模拟一个简单的返回，就在后台多起一个 Express 端口服务，极大地浪费了系统底噪内存（底噪占用 >130MB）。

*这促使我们在后期必须将架构全盘推翻，重构为 **Tauri 2.0 (Rust) + Win32 FFI + Windows 10/11 原生硬件级 WinRT OCR**，最终才把包体积从 93.6MB 压缩到了 3.37MB（降幅 96.4%），响应速度提升了 15 倍。*

## 📐 2. 同类自动化项目技术选型倾向

基于本次项目的教训，如果后续有类似满足 **“小包体 (Portable < 5MB)”、“冷启动毫秒级”、“物理硬件级高精度模拟”、“UI 科技美观”** 需求的 Windows 系统级工具，其技术选型倾向应做如下约束：

1. **核心逻辑全盘“去 JS/Node”**：
   * 严禁在 JavaScript 侧通过三方库执行截图、像素读取、按键模拟、进程捕获或 OCR。JS/TS 在物理资源交互、内存格式转码、硬件加速引擎通信上效率低下，且容易产生庞大的二进制依赖包（如 node-gyp）。
2. **前后端职责彻底解耦 (Rust 主宰系统)**：
   * 采用 **Tauri 2.0 (Rust) + Vite React** 架构。
   * 前端 Webview 仅作为**无状态呈现和配置收集面板**。所有的系统级轮询、窗口捕获、物理按键模拟、DCOMP 图层合成和 OCR 识别由 Rust 后端原生运行，通过异步单向 IPC 管道安全通信。
3. **零外部包体积的系统级 API 优先**：
   * 截图使用 GDI 位图直通（`BitBlt`），0 磁盘 I/O 损耗。
   * 识别使用 Windows 原生 WinRT 硬件 `OcrEngine`，直接借用系统底层计算资源。
   * 物理按键使用精确 padding 对齐的 Windows FFI `SendInput` API。

## 🤖 3. AI Agent 提示词与 Rules 宪法模版

为了在以后遇到同类型系统操作、透明合成、多显示器截图开发时**减少额度消耗、避免无效返工**，我们总结了这套 Rule，应在开发类似项目前直接喂给 AI 助手：

```markdown
# Windows 透明多窗口与屏幕截图开发宪法 (Tauri 2.0 & WebView2)

## 1. 窗口管理防卡死与防死锁规则
* 🚫 **严禁动态构建透明 Webview2 窗口**：绝对禁止在运行时通过工作线程调用 `WebviewWindowBuilder::build()` 来动态创建 `transparent(true)` 的大窗口。WebView2 的 DCOMP 透明合成机制容易因多线程句柄竞争导致系统级死锁挂起，引发白屏与交互彻底卡死。
*  **常驻后台单例模式**：所有透明辅助窗口（如截图选区遮罩、高亮描边窗口）必须在 `setup` 阶段（即程序启动前）一次性在主线程静默构建完毕，默认置为 `visible(false)` 隐身挂起。
* 🔄 **用 Show/Hide 代替 Build/Close**：运行时用户触发交互时，Rust 端在主线程通过 `.show()` 和 `.set_focus()` 瞬发唤醒已有窗口；选区完毕时执行 `.hide()` 隐藏，绝对禁止调用 `.close()` 销毁窗口。
* ⚡ **主线程调度律**：凡是涉及窗口操控（`.show()`, `.hide()`, `.set_size()`, `.set_position()`, `.close()`）的 Rust 端 IPC 命令，必须强制使用 `app_handle.run_on_main_thread(move || { ... })` 派发回 GUI 主线程顺序执行。

## 2. 多屏幕自适应与 DPI 缩放适配规则
* 🗺️ **多显示器独立窗口覆盖**：若要支持多屏幕（双屏/多屏）范围截图，严禁创建跨屏幕的单个巨大窗口。必须通过 `available_monitors()` 遍历所有可用物理屏幕，为每个屏幕在其物理起点上独立预建一个 `overlay_{i}` 窗口。
* 📏 **物理坐标与逻辑坐标转换**：
  * Tauri 窗口大小位置 (position / size) 和前端浏览器 DOM 在 Windows 上均采用 **逻辑像素 (Logical Pixels)**。
  * 物理屏幕抓图 (GDI API) 必须使用 **物理像素 (Physical Pixels)**。
  * **前端负责换算**：Rust 在预建窗口时，通过 `initialization_script` 向各屏幕窗口注入其专属的物理像素起点坐标 `physical_x/y` 和 `scale_factor` 缩放因子。
  * 前端选区画框结束后，直接在浏览器端依据局部逻辑像素计算出**跨屏幕的系统级绝对物理像素坐标 Rect** 返回给 Rust，Rust 接收后直通 GDI 截图，实现 100% 精准识别。

## 3. 彻底防进程残留 (防僵尸进程锁死 WebView2)
* 🔒 **防锁死机制**：在 Windows 上，当主窗口被销毁时，如果后台仍残留有隐藏辅助窗口，Tauri 进程不会自动退出。这会导致新进程因无法获取 WebView2 数据目录的读写排他锁而永久卡死白屏。
*  **双重退出钩子**：
  1. 将 UI 的关闭命令直接实现为 `app_handle.exit(0)`。
  2. 在 `tauri::Builder` 的 `.on_window_event` 钩子中，监听 `Destroyed`，一旦主窗口被销毁，立即强行执行 `window.app_handle().exit(0)`，确保绝对无残留进程。
```

## 🧠 4. 本次重构研发缺陷复盘与反思

本次重构开发过程中，我们遇到了“白屏卡死”和“重复返工”的情况，经深刻反思，其原因并非 User 提示词不准确，而是 **Agent 的问题定位不清晰与方案盲区**：

### 1. 对 Windows 窗口消息泵与 Webview 线程模型认识不足 (卡死根源)
* **Agent 盲区**：当用户首次反馈“点击识别范围后白屏未响应”时，Agent 将其误判为 React 前端异步状态加载时序（`getWindowLabel` 异步重绘）导致的状态冲突，从而在前端代码上进行了多次“缝补”。
* **实际病灶**：真实的死锁在于 **“在多线程异步 IPC Command 中直接构建透明 Webview 窗口，与正在活跃的主窗口图层合成器在底层 DXGI/DCOMP 锁上发生死锁”**。Agent 没能在第一步看清 Windows WebView2 在非主线程构建透明窗口的致命局限性，导致了方案方向性偏差，引起多次返工。

### 2. 多屏幕 DPI 缩放物理/逻辑像素混淆 (返工诱因)
* **Agent 盲区**：在缩放屏幕上点击无响应时，Agent 以为是窗口没有被 `.show()` 出来，没有预料到是因为把物理像素的 `width/height` 直接塞给了逻辑像素窗口 Builder，导致窗口巨大、把所有的 HTML 遮罩排布到了屏幕可视区之外。
* **总结**：由于 Agent 对 Windows 原生多显示器 DPI 缩放换算坐标系的时序逻辑未能一次性建立好“绝对物理坐标直通”的物理图景，导致前后端在逻辑/物理像素的转换界限上反复妥协、多处换算混淆，拉长了验证周期。

### 3. 残留进程判断失准导致调试“假死”
* **Agent 盲区**：当程序修复后，在覆盖覆盖测试时，旧有的僵尸进程仍在后台锁定 Webview2 目录，导致测试版双击打开仍然为白屏。Agent 未能及时通过进程管理器自检强杀后台残留，造成了调试结果的判断失准。

* **💡 结论与后续开发要求**：
  在遇到任何涉及透明窗口、屏幕截图、物理模拟等深层系统交互项目时，Agent 绝不能停留在前端 UI 和局部逻辑层进行尝试性修复。必须**第一时间从线程安全、物理像素/逻辑像素换算、以及系统进程残留锁这三个底层物理维度进行全局排查**，采用“常驻预加载”这一最稳定的桌面端经典架构，方可做到高效率、一次性安全交付。
